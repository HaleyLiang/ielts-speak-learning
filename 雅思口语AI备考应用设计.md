# ---

**雅思口语备考 Web 应用设计文档 (V1.0)**

## **1\. 产品概述**

本产品是一款移动端优先（Mobile-First）的 Web 应用，旨在利用大语言模型（LLM）协助雅思考生进行口语备考。核心亮点在于**用户自定义题库**、**AI引导生成个性化专属语料**以及**全真模拟考试交互**。第一阶段（V1.0）聚焦于核心功能闭环，支持文本与语音输入。

## **2\. 核心系统架构**

* **前端端 (Frontend):** 采用支持响应式和移动端优化的现代前端框架（如 React / Vue.js 配合 Tailwind CSS），页面交互以移动端单手操作习惯（如底部导航栏、大面积录音按钮）为主。  
* **后端服务 (Backend):** 轻量级 Node.js/Express 或 Python/FastAPI，主要负责处理业务逻辑、题库解析、大模型 API 请求转发（防止前端直接暴露请求逻辑，尽管 V1.0 允许用户自带 Key，后端转发便于统一组装 Prompt）。  
* **数据存储 (Database):** 考虑到题库结构的层级关系和用户自定义属性，推荐使用文档型数据库（如 MongoDB）或具备良好 JSON 支持的关系型数据库（如 PostgreSQL）。  
* **语音转文本 (STT):** V1.0 可直接调用浏览器原生的 Web Speech API 降低成本，或集成 OpenAI Whisper API 以提高中式英语的识别准确率。

**3\. 核心功能模块设计**

### **3.1 题库管理模块 (Question Bank Module)**

允许用户自主管理和更新雅思题库（通常每季更新），打破固定题库的限制。

* **题库结构:**  
  * **分类:** Part 1 / Part 2&3。  
  * **模块 (Module):** 对应具体的“话题 (Topic)”，例如 "Hometown", "Describe a person"。  
  * **问题 (Question):** 话题下的具体提问。  
  * **个人回答 (Personal Answer):** 挂载在每个问题下，存储用户通过 AI 引导生成的专属答案。  
* **功能交互:**  
  * **上传/导入:** 支持通过 JSON/CSV 文件批量导入题库，或在界面上手动添加/编辑话题和问题。  
  * **展示与管理:** 列表展示所有话题，以标签形式区分“已准备语料”和“未准备语料”的话题。

### **3.2 AI 对话模块 (AI Dialogue Module)**

整个应用的核心交互区，高度依赖 Prompt 工程。

#### **功能 A：练习模式 (Practice Mode \- 语料定制与定点练习)**

**业务流转:**

1. **随机抽题:** 系统从题库中随机抽取一个 Part 1 话题和一个 Part 2&3 话题组合。  
2. **语料定制 (Prompt Engineering 核心):**  
   * 界面进入“语料打磨”阶段。  
   * LLM 扮演“雅思口语外教”，针对抽中的问题，先询问用户的真实经历或想法（例如：“关于 Hometown，你最喜欢它的什么？用中文或简单的英文关键词告诉我”）。  
   * 接收用户输入后，LLM 将其扩写、润色为符合雅思高分标准（地道表达、复杂句式、逻辑连词）的个人专属英文答案。  
3. **数据落库:** 用户确认后，将生成的答案自动保存至题库该问题下的 Personal Answer 字段。  
4. **对练环节:** 界面切换至对话框模式。AI 提出刚才准备过的问题，用户通过**语音输入**作答，系统将用户的录音转录为文本，并与刚刚生成的 Personal Answer 进行比对，给出流利度和词汇使用的纠错建议。

#### **功能 B：模拟考试 (Mock Exam Mode \- 全真模拟与评分)**

**业务流转:**

1. **流程控制:** 严格按照雅思真实考试时长和流程进行：  
   * ID Check & Part 1 (约 4-5 分钟，3个话题)。  
   * Part 2 (1分钟准备，2分钟作答，支持在页面上做 Note-taking)。  
   * Part 3 (约 4-5 分钟，基于 Part 2 话题的深入探讨)。  
2. **动态提问:** LLM 扮演严谨的“雅思考官”。根据预设的 System Prompt，它不会对用户的回答做即时纠错，而是根据用户的回答内容，动态抛出下一个问题或进行追问（尤其是 Part 3）。  
3. **综合评分:** 考试结束后，后端将整个对话的 Transcript（转录文本）打包发送给 LLM。根据雅思口语四大评分标准（流利性与连贯性、词汇多样性、语法多样性及准确性、发音-基于转录文本的预估）输出一份详细的 Report，包含预估分数、高光词汇和改进建议。

### **3.3 用户设置模块 (User Settings Module) *\[更新版\]***

* **API 与模型配置:** 提供输入框供用户填入自己的大模型 API Key，并允许用户从下拉列表中选择要调用的具体模型（如 gpt-4o-mini, gpt-3.5-turbo 等），将控制权和成本管理交还给用户。  
* **目标分数设定 (Target Score Configuration):** \* 提供一个直观的滑动条 (Slider) 或下拉菜单，允许用户选择自己的雅思口语目标分数（可选范围建议设为：5.5, 6.0, 6.5, 7.0, 7.5, 8.0+）。  
  * **技术实现:** 该设定作为全局变量（如 target\_score）持久化存储在前端。在发起每次 API 请求时，系统会将此变量动态注入到后端的 Prompt 模板中。  
* **本地缓存:** 所有配置（API Key、模型偏好、目标分数）均保存在浏览器的 LocalStorage 中，保障用户隐私并实现开箱即用。

**4\. 核心 Prompt 工程设计 (示例) *\[更新版\]***

在这一版本中，Prompt 模板引入了动态变量 \[Target Score\]，确保 AI 的输出颗粒度与用户需求完美契合。

* **语料定制 Prompt (练习模式 \- 引入动态分数):**

  "你是一位资深的雅思口语前考官。现在我们要为学生定制雅思回答。当前题目是：\[Insert Question\]。学生的真实想法/经历是：\[User Input\]。  
  请根据学生的想法，创作一个严格符合雅思口语 **\[Target Score\]** 分标准的英文回答。  
  核心要求：

  * **词汇与语法定级:** 词汇难度和语法复杂度必须严格锚定 **\[Target Score\]** 分的真实水平。如果目标分低于7.0，请避免使用过于生僻或华丽的大词，多用自然地道的词组搭配（collocations）；如果目标分大于等于7.0，请适当增加复杂句式和高级词汇。  
  * **口语化表达:** 保持自然的口语交流感，避免机器味或书面背诵感。  
  * **长度限制:** 控制在 \[根据Part不同动态设定的字数\] 以内。

请直接输出英文回答，并在下方附带三个匹配该分数段的“提分重点词汇/短语”解析。"

* **考官扮演 Prompt (模考模式 \- 保持不变):**"你现在是雅思口语考官。正在进行 \[Part X\] 的考试。考生的上一句回答是：\[Transcript\]。请评估是否需要根据其回答进行追问，或者进入题库中的下一个问题：\[Next Question\]。请仅输出你作为考官应该说出的英文句子，不要包含任何评分、反馈或多余的动作描写。"  
* **综合评分 Prompt (模考结束 \- 结合目标分给出反馈):**

  "这是一场完整的雅思口语模拟考试对话记录：\[Full Transcript\]。**该考生的预设目标分数是 \[Target Score\] 分。**  
  请根据雅思口语四项评分标准（FC, LR, GRA, PR \- 注意PR主要基于文本逻辑预估）进行严格打分。输出 JSON 格式，必须包含以下字段：

  * overallBand: 预估总分。  
  * gapAnalysis: 简要说明当前表现距离目标分数 **\[Target Score\]** 还有多大差距，或是否已达标。  
  * part1Feedback, part2Feedback, part3Feedback: 各部分的具体反馈。  
  * grammarCorrections: 语法纠错列表。  
  * betterExpressions: 提供从考生当前水平向 **\[Target Score\]** 分迈进的地道表达替换建议。"

**5\. UI/UX 布局建议 (移动端视角)**

* **主导航 (Bottom Tab Bar):**  
  * **🎙️ 练习 (Practice):** 核心工作区，顶部提供 Switch 切换“单项练习”与“全真模考”。中心是大大的麦克风按钮和对话气泡区。  
  * **📚 题库 (Bank):** 手风琴式的列表（Accordion），按 Part 和 Topic 展开，直观看到哪些话题的“进度条”已填满（即已生成专属答案）。  
  * **⚙️ 设置 (Settings):** API Key 输入框、模型选择、以及清除本地数据的选项。  
* **交互细节:**  
  * 在 Part 2 模考时，屏幕上半部分锁定展示 Cue Card（话题卡），下半部分提供倒计时 UI（1分钟准备，2分钟录音）。  
  * 语音输入时需有明显的波形动画反馈，消除用户的等待焦虑。
